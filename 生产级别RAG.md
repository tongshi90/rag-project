# 文档处理阶段
## 功能介绍
	将文档拆分成文本、图片、表格分别进行处理

	图片：每个图片是一个chunk，内容包含OCR文本、上下文图片描述等信息

	表格：每一行数据作为一个chunk，内容需要转换成标题:内容的键值对格式

	文本：每个段落作为一个chunk(核心是怎么区分哪段内容是一个段落???)，只要是同一个段落，就算是分页也需要在一个段落中。如果段落太大无法放到一分片中的时候，可以放到多个分片中，但是需要有内容重叠。

	所有分片都需要携带文档名、文档层级(章、节、标题)、页码、图片/表格所在的位置、类型(表格、图片、文本)，chunk需要使用同一格式返回

## 业务实现方式
### 1、数据提取
使用pdfplumber依赖库，读取pdf文件。提取文本信息（分页数据也都先放在一起，后续再做切分，这样避免了分页导致内容被切分成两个chunk的问题）、提取表格信息 （每一行数据作为一条记录，并将格式整理成标题：内容这种键值对的形式。注意要考虑表格被切分到两页之中的时候，表头需要用前面那一页甚至更前页中的表头数据）、提取图片信息（每张图片是一个记录，并使用OCR模型转换成文本信息）。

### 2、文本数据拆分
上面的文本数据，是整个pdf的文本都放一起的，一般对于标准的pdf格式而言，建议按照段落、章节等进行拆分（如果段落、章节内容太多的话，可以拆分成几个片，但是这种就需要有数据重叠。且拆分的时候需要按照标点符号进行整句话级别的拆分）

## 功能介绍

	{

	  "doc_id": "doc_001",          # 所属文档的唯一ID，用于关联原始文档（如PDF文件ID）

	  "chunk_id": "table_12_row_3", # 切片的唯一ID，全局唯一（如表格_页码_行号），用于定位/更新/删除

	  "start_idx": 156,             # 切片在原始文档中的起始字符位置（可选，文本切片用）

	  "end_idx": 218,               # 切片在原始文档中的结束字符位置（可选，文本切片用）

	  "confidence": 0.92,         # 切片生成的置信度（如表格识别/文本提取的准确率），过滤<0.7的低质量切片

	  "length": 62,                 # 切片内容长度（token数/字符数），过滤过短/过长切片

# 向量化
## 功能介绍
	embedding失败后重试，长文本要自动摘要之后再embedding

	多向量策略，也就是做多套embedding(每套都单独存储一套向量结果信息)。注意：如果做多套之后，之后向量检索的时候，需要对所有套结果都计算，最终按照权重等方式计算出每个分片的实际距离信息，再拿到距离最小的top_N

# 分片检索
## 功能介绍
本质上就是向量检索+重排序+上下文拼接(LLM中直接使用这个作为上下文内容)

**向量检索阶段：**除了单纯的向量检索外，还需要结合关键词匹配(分别占一定权重，甚至关键字的权重要更高)，最终得到距离分数，得到距离最近的top_n(top_n一般是top_k的5-10倍，使用余弦相似度计算向量距离)

	**重排序：**使用前面得到的top_n，计算这些分片的文本相似度。加权因素还需要考虑以下几点(chunk类型权重，参数类的权重要大于描述类；文档权威程度；文档新旧程度；OCR置信度等)	，得到top_k(一般使用使用预训练重排序模型的交叉注意力语义推理方式，相较余弦相似度，有更细粒的文本对语义匹配，但是执行速度更慢)

	**上下文拼接：**不仅仅是将top_k分片简单拼接，还需要考虑同一个文本优先合并、同一个表格合并、去重等。并需要注意token数量不要超过LLM模型能承受的上下文极限
